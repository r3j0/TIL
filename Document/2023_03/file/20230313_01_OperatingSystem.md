Operating System Concepts

### 1.2 Computer-System Organization

![20230313_01_01](https://github.com/r3j0/TIL/blob/main/Document/2023_03/file/img/20230313_01_01.png)

현대 범용 컴퓨터 시스템은 (하나 이상의 "`CPU`")와 (구성요소와 공유 메모리 사이의 액세스를 제공하는 `공통 버스`를 통해 연결된 "`여러 장치 컨트롤러`")로 구성된다.  
> 장치 컨트롤러
> - 일부 로컬 버퍼 저장소와 특수 목적 레지스터 집합을 유지 관리
> - 제어하는 주변 장치와 로컬 버퍼 저장소 간에 데이터를 이동

일반적으로 운영체제에는 각 장치 컨트롤러마다 `장치 드라이버`가 있다. 장치 드라이버는 장치 컨트롤러의 작동을 잘 알고 있고 나머지 장치에 대한 일관된 인터페이스를 제공한다. CPU와 장치 컨트롤러는 병렬로 실행되어 메모리 사이클을 놓고 경쟁한다. 공유 메모리를 질서 있게 액세스하기 위해 메모리 컨트롤러는 메모리에 대한 액세스를 동기화한다.

### 1.2.1 Interrupts
- 일반적인 컴퓨터 작업
입출력 작업 시작  
-> 장치 드라이버는 장치 컨트롤러의 적절한 레지스터에 값을 적재  
-> 장치 컨트롤러는 레지스터의 내용을 검사해 수행할 작업 결정 ( ex. 키보드에서 문자 읽기 )  
-> 컨트롤러가 장치에서 로컬 버퍼로 데이터 전송 시작  
-> 데이터 전송 완료 시 장치 컨트롤러가 장치 드라이버에게 작업이 완료됨을 알림  
-> 장치 드라이버는 읽기 요청이면 데이터나 데이터에 대한 포인터를 반환하고 운영체제의 다른 부분에 제어를 넘김. 다른 작업의 경우 "쓰기 완료", "장치 사용 중"과 같은 상태 정보 반환

이 때 컨트롤러는 장치 드라이버에게 작업이 완료됐다는 사실은 `인터럽트`를 통해 이루어진다.  

하드웨어는 어느 순간이든 시스템 버스를 통해 CPU에 신호를 보내 인터럽트를 발생시킨다. ( 컴퓨터 시스템의 많은 버스 중 시스템 버스 - 주요 구성요소 사이의 주요 통신 경로 ) 인터럽트는 운영체제와 하드웨어 상호 작용의 핵심 부분이다.  
CPU가 인터럽트되면, CPU는 하던 일을 중단하고 즉시 고정된 위치로 실행을 옮긴다. ( 고정된 위치 : 인터럽트를 위한 서비스 루틴이 위치한 시작 주소 ) 그리고 인터럽트 서비스 루틴이 실행된다. 서비스 루틴을 완료하면 인터럽트 되었던 연산을 재개한다.

![20230313_01_02](https://github.com/r3j0/TIL/blob/main/Document/2023_03/file/img/20230313_01_02.png)

인터럽트는 적절한 서비스 루틴으로 제어를 전달한다. (적절한 서비스 루틴을 수행한다.) 이러한 전달을 관리하기 위해 인터럽트 정보를 조사하는 일반적인 루틴을 호출한다. (직접적인 방법) 이 루틴은 인터럽트 고유의 핸들러를 호출한다. 
하지만 인터럽트는 매우 빈번하게 발생하기 때문에 빠르게 처리되어야 한다. 이에 맞는 속도를 제공하기 위해 인터럽트 루틴에 대한 포인터들의 테이블(하위 메모리에 저장. 여러 장치에 대한 인터럽트 서비스 루틴의 주소가 들어있음)을 대신 이용할 수 있다. (간접적인 방법. 중간 루틴을 둘 필요 없음)  

인터럽트가 요청되면, 인터럽트를 유발한 장치를 위한 인터럽트 서비스 루틴의 주소를 제공하기 위해 이 주소의 배열인 `인터럽트 벡터`가 인터럽트 요청과 함께 주어진 고유의 유일한 장치 번호로 색인된다.  
인터럽트 구조는 또한 인터럽트된 모든 정보를 저장해야 인터럽트를 처리한 후 이 정보를 복원할 수 있다. 만약 인터럽트 루틴이 처리기의 상태를 변경할 필요가 있다면 ( ex. 레지스터의 값을 변경할 필요 ), 인터럽트 루틴은 반드시 명시적으로 현재의 상태를 저장하고 복귀 전에 상태를 복원해야 한다.  
인터럽트 서비스 이후 저장된 복귀 주소를 프로그램 카운터에 적재하고, 중단된 연산이 인터럽트가 발생되지 않았던 것처럼 다시 시작된다.  

CPU 하드웨어에는 `인터럽트 요청 라인 (interrupt request line)` 이 있는데, 하나의 명령어의 실행을 완료할 때마다 CPU가 이 선을 감지한다. 컨트롤러가 인터럽트 요청 라인에 신호를 보낸 것을 CPU가 감지하면, 인터럽트 번호를 읽어 인터럽트 벡터의 인덱스로 사용하여 `인터럽트 핸들러 루틴 (interrupt-handler routine)`으로 점프한다. 해당 인덱스 주소에서 실행을 시작한다.  
인터럽트 처리기가 하는 일  
- 작업 중에 변경될 상태를 저장
- 인터럽트 원인을 확인
- 필요한 처리를 수행
- 상태 복원을 수행
- return_from_interrupt 명령어를 실행해 CPU를 인터럽트 전 상태로 되돌림

장치 컨트롤러가 인터럽트 요청 라인에 신호를 보내 인터럽트를 `발생(raise)`시키고, CPU가 인터럽트를 `포착(catch)`하여 인터럽트 핸들러로 `디스패치(dispatch)`하고 핸들러는 장치를 서비스하여 인터럽트를 `지운다(clear)`.

![20230313_01_03](https://github.com/r3j0/TIL/blob/main/Document/2023_03/file/img/20230313_01_03.png)

이 인터럽트 기법은 장치 컨트롤러가 서비스할 준비가 될 때와 같은 비동기 이벤트에 CPU가 대응할 수 있게 해준다.  
최신 운영체제에서는 더욱 정교한 인터럽트 처리 기능이 필요하다. ( 이 기능은 CPU 및 인터럽트 컨트롤러 하드웨어에 의해 제공된다. )

1. 중요한 처리 중 인터럽트 처리를 연기할 수 있어야 한다.
2. 장치의 적절한 인터럽트 핸들러로 효율적으로 디스패치 할 방법이 필요하다.
3. 운영체제가 우선순위에 따라 인터럽트를 구분하고 적절한 긴급도로 대응할 수 있도록 다단계 인터럽트가 필요하다.

대부분 CPU에는 2개의 인터럽트 요청 라인이 있다.
1. 마스크 불가능 인터럽트 (nonmaskable interrupt) - 복구할 수 없는 메모리 오류와 같은 이벤트를 위해 예약됨
2. 마스킹 가능 (maskable) - 장치 컨트롤러가 서비스를 요청하기 위해 사용. 중요한 명령 시퀀스를 실행하기 전에 CPU에 의해 꺼질 수 있음.

벡터 방식 인터럽트 기법의 목적 : 서비스가 필요한 장치를 결정하기 위해 하나의 인터럽트 핸들러가 가능한 모든 인터럽트 소스를 검색할 필요를 줄이기 위한 것.  
실제로 컴퓨터에는 인터럽트 벡터의 주소 개수보다 많은 장치 (인터럽트 처리기)가 있다. 이 문제를 해결하는 일반적인 방법은 `인터럽트 체인`을 사용하는 것이다.  
인터럽트 벡터의 각 원소는 인터럽트 핸들러 리스트의 헤드를 가리킨다. 인터럽트가 발생하면 요청을 처리할 수 있는 핸들러가 발견될 때까지 상응하는 리스트의 핸들러가 하나씩 호출된다. 이러한 구조는 큰 크기의 인터럽트 테이블의 오버헤드와 하나의 인터럽트 핸들러로 디스패치하는 비효율성의 절충안이다. 
Intel 프로세서는 마스크 불가능한 0에서 31까지의 이벤트는 다양한 오류 조건을 알리는 데 사용되고, 마스크 가능한 32에서 255까지의 이벤트는 장치가 생성한 인터럽트 같은 그 외 인터럽트를 처리하기 위해 사용된다.  

인터럽트 기법은 `인터럽트 우선순위 레벨 (interrupt priority level)` 을 구현한다. 이를 통해 CPU는 모든 인터럽트를 마스킹하지 않고도 우선순위가 낮은 인터럽트 처리를 연기할 수 있고, 우선순위가 높은 인터럽트가 실행을 선점할 수 있다.

장치 컨트롤러 및 하드웨어 오류로 인해 발생하는 인터럽트는 최신 운영체제에서 비동기 이벤트를 처리하기 위해 사용된다. 가장 긴급한 작업을 먼저 수행하기 위해 최신 컴퓨터는 인터럽트 우선순위 시스템을 사용한다. 인터럽트는 시간에 민감한 처리에 빈번하게 사용되므로 시스템 성능을 좋게 하려면 효율적인 인터럽트 처리가 필요하다.

### 1.2.2 Storage Structure
CPU는 메모리에서만 명령 적재가 가능하다. -> 실행하려면 프로그램을 먼저 메모리에 적재해야 한다.  
범용 컴퓨터는 프로그램 대부분을 `메인 메모리 (random-access memory / RAM), (dynamic random-access memory (DRAM))`라 불리는 재기록 가능한 메모리에서 가져온다.
컴퓨터 전원을 켤 때 가장 먼저 실행되는 프로그램은 `부트스트랩 프로그램`이며 운영체제를 적재한다. RAM은 `휘발성 (전원이 꺼지거나 손실될 때 내용 손실)`이므로 부트스트랩 프로그램을 유지하지 못한다. 대신 전기적으로 소거 가능한 프로그램 읽기 전용 메모리 (EEPROM) (자주 변경은 불가, 속도가 느려 주로 사용되지 않는 프로그램과 데이터 저장) 및 기타 형태의 `펌웨어 (쓰기 작업이 자주 발생하지 않는 비휘발성 저장장치)`를 사용한다.  

모든 형태의 메모리는 바이트의 배열을 제공한다. ( 각 바이트는 자신의 주소를 가진다. ) 특정 메모리 주소들에 대한 적재 (load), 저장 (store) 명령이 이루어진다.
- 적재 명령 : 메인 메모리로부터 CPU 내부의 레지스터로 한 바이트 또는 한 워드를 옮긴다.
- 저장 명령 : 레지스터의 내용을 메인 메모리로 옮긴다.
이 명령 외에 CPU는 프로그램 카운터에 저장된 위치부터 실행하기 위해 메인 메모리에서 명령을 자동으로 적재한다.
